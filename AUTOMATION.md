# 주간 배포와 자동답변 운영 경계

## 다음 공개 후보에 포함된 자동화

v0.1.2 후보가 정확한 revision 승인을 받아 기본 브랜치에 공개된 뒤부터, 매주
금요일 21:17(Asia/Seoul)에 GitHub Actions가 공개 저장소 안전검사와 배포 준비
상태를 읽기 전용으로 확인하도록 설정되어 있다. 현재 공개 v0.1.1에는 아직 이
워크플로가 없다. 정각 부하를 피하기 위해 17분으로 설정했다. 이 작업은 파일,
릴리스, 이슈, 홈페이지, 데이터베이스를 수정하지 않는다.

GitHub의 예약 워크플로는 기본 브랜치에서 실행되며 부하가 높으면 늦어지거나
누락될 수 있고, 공개 저장소가 60일 동안 비활성 상태면 예약 실행이 꺼질 수 있다.
따라서 예약 결과 자체를 정확한 배포 시각 보장으로 사용하지 않는다.

근거: [GitHub Actions schedule 이벤트](https://docs.github.com/en/actions/reference/workflows-and-actions/events-that-trigger-workflows#schedule)

## 주간 패치 공개로 가는 단계

1. 대상 게임 하나를 지정해 수동 릴리스 스프린트 3회를 완료한다.
2. 매번 R0~R5, 클린 적용, ROM 미포함, ZIP 해시, 사용자 승인을 기록한다.
3. 자동화는 승인된 원본에서 패키지 후보와 체크섬을 만들고 초안 릴리스까지만 준비한다.
4. GitHub Environment의 required reviewer가 정확한 패키지 해시를 승인한 뒤에만 공개한다.
5. 두 번 연속 실패하면 중단하고 kill switch를 유지한다.

필수 사람 결정인 용어·말투·권리·정확한 해시 배포 승인은 자동화하지 않는다.
GitHub Environment는 승인 전에는 해당 환경의 secret을 작업에 제공하지 않도록
구성할 수 있다.

근거: [GitHub 배포 환경과 보호 규칙](https://docs.github.com/en/actions/reference/workflows-and-actions/deployments-and-environments)

## 이용자 글과 답변 데이터베이스

D1은 다음 상태를 분리해 저장한다.

- `visitor_posts`: 로그인 사용자의 글, 공개 닉네임, 동의 revision, 90일 처리 기한, 철회키 해시
- `feedback_rate_limits`: 도배 방지용 비가역 식별값과 시간창별 요청 수, 2시간 사용 기한
- `reply_jobs`: 자동답변 초안 생성 대기열, 재시도 횟수, 실패 코드
- `feedback_replies`: 사람·템플릿·AI가 만든 답변과 검토·공개 상태
- `moderation_events`: 승인·거부·차단 근거
- `automation_runs`: 읽기 전용 점검과 향후 초안 생성의 실행 증거

원문 이메일은 저장하지 않는다. 런타임 비밀값과 이메일을 함께 해시한 비가역
식별값은 별도 표에서 도배 방지에만 사용하고 2시간 뒤에는 더 이상 조회·판단에
사용하지 않는다. 글과 공개 닉네임도 90일이 지나면 조회·답변·공개 대상에서 즉시
제외한다. 현재 코드는 새 접수 때 만료 행을 정리하지만, 새 접수가 없을 때의 물리
삭제를 보장하는 반복 정리 작업은 아직 꺼져 있다. 따라서 물리 삭제 게이트가 별도
승인되기 전까지 `FEEDBACK_WRITE_ENABLED=false`를 유지하고 실제 글 접수를 열지
않는다.

접수 때 철회키 원문은 브라우저에만 전달하고 서버에는 해시만 저장한다. 90일 안의
철회키는 임의 개수 제한으로 버리지 않으며, 브라우저 저장 실패 시 화면에 키를
표시해 즉시 복사하거나 글을 철회할 수 있게 한다. 브라우저 데이터나 철회키를 잃으면
서버에서 복구할 수 없다. 글은 시간당 3건으로 제한하고 연락처가 포함된 글, 링크가
많은 글, ROM·디스크·압축 파일 직접 링크를 거부한다.

접수된 글은 `pending`으로만 저장되며 답변 작업을 동시에 만들지 않는다. 사람이
글을 `approved`로 바꾼 뒤에만 `queued → drafted → approved → published` 답변
절차를 시작할 수 있다. 현재 답변 생성기와 자동 공개는 모두 꺼져 있다. AI 제공자,
비용 상한, 금지 주제와 답변 정책이 승인되기 전에는 자동답변을 생성하거나 공개하지
않는다.

## D1 배포 전 필수 게이트

1. Sites에 D1 논리 바인딩 `DB`를 만들되 `FEEDBACK_WRITE_ENABLED=false`를 유지한다.
2. `drizzle/`의 단일 baseline migration을 새 D1에 적용하고 6개 테이블을 확인한다.
3. 32바이트 이상의 무작위 `FEEDBACK_ID_PEPPER`를 호스팅 비밀값으로 설정하고 코드의 길이 검사를 통과한다.
4. `.env.example`에는 빈 변수 이름만 두며 실제 값은 커밋하거나 로그에 남기지 않는다.
5. 로그인 접수·3건 제한·철회·공개 글 읽기와 만료 행 물리 삭제의 실제 환경 검사를 통과한다.
6. 위 증거가 묶인 정확한 revision과 호스팅 상태 변경을 사용자가 별도로 승인한 뒤에만 `FEEDBACK_WRITE_ENABLED=true`로 바꾼다.

`FEEDBACK_WRITE_ENABLED=true`는 소스 배포에 따라 자동으로 켜지지 않는 별도 공개
쓰기 승인이다. v0.1.2 코드·D1 기반을 배포하더라도 기본값은 `false`다.
